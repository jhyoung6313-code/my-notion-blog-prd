## 이 문서의 목적

[PRD.md](./PRD.md)가 **"무엇을(What)"** 만들지 정의한다면,
이 ROADMAP은 **"어떤 순서로(How/When)"** 만들지 정의한다.

기능부터 하나씩 완성하는 방식(❌ 목록 → 상세 → 필터 → 검색)은
공통 코드를 매번 중복 작성하게 만들어 유지보수를 어렵게 한다.
대신 **견고한 골격과 공통 모듈을 먼저 세우고, 개별 기능을 그 위에 올린다.**

```
Phase 1 (골격)  →  Phase 2 (공통)  →  Phase 3~4 (개별 기능)  →  Phase 5 (최적화/배포)
   기반 환경         재사용 코드          공통 위에 구현            품질 향상
```

## Phase 1: 프로젝트 초기 설정 (골격 구축)

> **예상 소요** 1~2일
> **이 단계의 목표** 코드를 한 줄도 안 짠 상태에서, 기능을 올릴 수 있는 **빈 골격**을 만든다.

### 할 작업
- [ ] Next.js 15 (App Router) + TypeScript 프로젝트 생성
- [ ] Tailwind CSS + shadcn/ui + Lucide React 설치·설정
- [ ] 기본 레이아웃 구조 생성 (`app/layout.tsx`, 헤더/푸터 자리)
- [ ] Notion Integration 생성 및 데이터베이스 연결
- [ ] 환경 변수 설정 (`.env.local`: `NOTION_API_KEY`, `NOTION_DATABASE_ID`)
- [ ] `.gitignore`에 `.env.local` 포함 확인 (키 노출 방지)
- [ ] 디렉터리 구조 확정 (`app/`, `lib/`, `components/`, `types/`)

### 왜 이 순서인가?
> 견고한 기반 없이 기능부터 만들면, 폴더 구조와 환경 설정이 흔들릴 때마다 이미 만든 기능을 전부 다시 손봐야 한다. **변경 비용이 가장 큰 결정(프레임워크·구조·연동)을 가장 먼저** 확정한다.

### 완료 기준
- `npm run dev`로 빈 페이지가 정상 구동된다.
- Notion API에 테스트 요청을 보내 **응답 200을 한 번 이상 확인**했다.
- 환경 변수가 `.env.local`에만 있고 Git에 추적되지 않는다.

## Phase 2: 공통 모듈/컴포넌트 개발 (공통 기반)

> **예상 소요** 2~3일
> **이 단계의 목표** 모든 페이지가 **공유해서 쓸 코드**를 먼저 만든다. 여기서 만든 것을 Phase 3~4가 재사용한다.

### 할 작업
- [ ] **공통 타입 정의** (`types/post.ts`): `Post`, `Category`, `Tag` 인터페이스
- [ ] **Notion 데이터 페칭 레이어** (`lib/notion.ts`)
  - [ ] `getPublishedPosts()` — 발행된 글 목록 (`Status = 발행됨`, 발행일 내림차순)
  - [ ] `getPostBySlug(slug)` — slug로 단건 조회
  - [ ] `getPostContent(pageId)` — 본문 블록 조회
  - [ ] Notion 응답 → `Post` 타입 매핑 유틸 (`mapPageToPost`)
- [ ] **본문 렌더링 모듈**: Notion 블록 → JSX 변환 (`react-notion-x` / `notion-to-md`)
- [ ] **공통 UI 컴포넌트** (`components/`)
  - [ ] `Header`, `Footer`, `Layout`
  - [ ] `PostCard` (목록·필터 페이지에서 공용)
  - [ ] `CategoryTabs`, `TagBadge`
- [ ] 상수 정의 (`lib/constants.ts`): ISR `revalidate` 값, 카테고리 enum 등 매직 넘버 제거

### 왜 이 순서인가?
> 목록 페이지, 상세 페이지, 카테고리 페이지는 **모두 같은 Notion 조회 함수와 같은 카드 컴포넌트를 쓴다**. 이 공통 코드를 먼저 만들지 않고 페이지마다 따로 작성하면 → 중복 코드 발생 → 한 곳을 고치면 다른 곳이 깨진다. **재사용 단위를 먼저 못 박아야** 이후 개별 기능 개발 속도가 빨라진다.

### 완료 기준
- `lib/notion.ts`의 함수를 단독 호출해 **콘솔에 실제 글 데이터가 찍힌다.**
- `PostCard`에 더미 데이터를 넣으면 카드가 정상 렌더링된다.
- 모든 데이터 접근이 `lib/notion.ts`를 거치며, 페이지에서 직접 Notion API를 호출하지 않는다.

## Phase 3: 핵심 기능 개발 (MVP)

> **예상 소요** 3~4일
> **이 단계의 목표** Phase 2의 공통 모듈 위에 **블로그의 본질("쓴 글이 보이고, 클릭하면 읽힌다")** 을 올린다.

### 할 작업
- [ ] **홈 / 글 목록 페이지** `/` (F-01) — `getPublishedPosts()` + `PostCard` 그리드
- [ ] **글 상세 페이지** `/posts/[slug]` (F-02) — `getPostBySlug` + 본문 렌더링
- [ ] Notion 본문 블록 렌더링 (텍스트·이미지·코드 블록)
- [ ] **카테고리 필터** (F-03) — `CategoryTabs`로 목록 좁히기
- [ ] **반응형 기본 레이아웃** (F-06) — 모바일/태블릿/데스크톱 그리드
- [ ] ISR 설정 (`revalidate: 60`)로 데이터 캐싱

### 왜 이 순서인가?
> 검색·태그·다크모드보다 **목록과 상세가 먼저 동작해야** 블로그라고 부를 수 있다. 가장 핵심적이고 사용자 가치가 높은 기능을, 이미 검증된 공통 모듈 위에 빠르게 구현한다. 이 단계 종료 시점이 곧 **MVP 완성**이다.

### 완료 기준
- Notion에 글을 발행하면 홈 목록에 카드로 나타난다.
- 카드를 클릭하면 상세 페이지에서 본문(텍스트·이미지·코드)이 정상 렌더링된다.
- 카테고리 탭으로 글이 필터링된다.
- 모바일·데스크톱에서 레이아웃이 깨지지 않는다.

## Phase 4: 추가 기능 개발 (P1 이후)

> **예상 소요** 2~3일
> **이 단계의 목표** MVP가 동작하는 상태에서 **부가 가치 기능**을 더한다.

### 할 작업
- [ ] **태그 표시·필터** (F-04) — `/tags/[tag]` (공통 `TagBadge` 재사용)
- [ ] **검색** (F-05) — 제목·요약 클라이언트 사이드 부분 일치 검색
- [ ] **SEO 메타 태그** (F-08) — 글별 OG 태그·메타 디스크립션 (`generateMetadata`)
- [ ] **다크 모드** (F-07) — 라이트/다크 토글 + localStorage 영속화

### 왜 이 순서인가?
> 이 기능들은 **핵심 기능이 없으면 의미가 없다**(검색할 글이 없으면 검색 기능도 무용). 핵심이 안정화된 뒤 추가해야 우선순위가 명확하고, Phase 2의 공통 컴포넌트(`TagBadge`, `PostCard`)를 그대로 재사용하므로 개발 비용이 적다.

### 완료 기준
- 태그 클릭 시 해당 태그 글만 표시된다.
- 검색창에 키워드 입력 시 제목·요약이 일치하는 글이 필터링된다.
- 각 글 상세 페이지가 고유한 OG 태그·메타 디스크립션을 갖는다.
- 다크 모드 토글이 새로고침 후에도 유지된다.

## Phase 5: 최적화 및 배포

> **예상 소요** 1~2일
> **이 단계의 목표** 완성된 기능의 **품질을 끌어올리고 공개**한다.

### 할 작업
- [ ] 성능 최적화 — 이미지 최적화(`next/image`), 폰트 최적화, 번들 점검
- [ ] 반응형 디자인 마감 — 엣지 케이스(긴 제목, 이미지 없는 글) 점검
- [ ] 접근성 점검 — 시맨틱 태그, alt 텍스트, 키보드 내비게이션
- [ ] Notion API 레이트리밋 대응 점검 (ISR 캐시 동작 확인)
- [ ] **Vercel 배포** — 프로젝트 연결, 환경 변수 등록, ISR 설정
- [ ] 배포 후 Lighthouse 측정 및 보정

### 왜 이 순서인가?
> 최적화는 **기능이 완성된 후에 해야** 한다. 미완성 기능을 미리 최적화하면 구조가 바뀔 때 최적화 작업이 버려진다("이른 최적화는 만악의 근원"). 모든 기능이 동작하는 상태를 기준으로 측정·개선하고 배포한다.

### 완료 기준
- Notion에서 글 발행 시 **1분 이내** 사이트에 반영된다. (성공 지표)
- 홈·상세 페이지 **Lighthouse 성능 90점 이상**. (성공 지표)
- 공개 URL에서 코드 수정 없이 **Notion 작성만으로** 새 글이 발행된다. (성공 지표)

```
[Phase 1: 골격]
  Next.js · Tailwind · Notion 연동 환경
        │  (기반이 없으면 아무 기능도 못 올림)
        ▼

        │  (모든 페이지가 이 모듈을 재사용)
        ▼
[Phase 3: 핵심 기능 = MVP]
  글 목록 · 글 상세 · 카테고리 필터

  태그 · 검색 · SEO · 다크모드

        ▼
[Phase 5: 최적화 및 배포]
  성능 · 반응형 마감 · Vercel 배포
```
